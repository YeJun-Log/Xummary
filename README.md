# Xummary 
: AI 기반 실시간 경제 분석 및 포트폴리오 리밸런싱 서비스

## 1. 프로젝트 개요 (Overview)
* **프로젝트명:** Xummary
* **핵심 가치:** 흩어져 있는 전문가들의 파편화된 경제 지식을 AI로 통합 분석하여 개인화된 투자 전략으로 변환.
* **주요 기능:**
  * Nitter(X 우회)를 활용한 실시간 경제 전문가 트윗 및 이미지 수집.
  * Gemini 모델을 통한 차트 및 텍스트 교차 분석.
  * 사용자의 투자 포트폴리오를 분석하여 맞춤형 리밸런싱 전략 제안. (VIP 전용)
  * GitHub Actions를 활용한 매주 자동화 리포트 발송.
* **기술적 의의:** 멀티모달 LLM과 지수 백오프 기반의 재시도 로직을 결합하여, 불안정한 외부 스크래핑 환경 및 API 서버 과부하 속에서도 중단 없이 동작하는 무결점 데이터 파이프라인 구축.

## 2. 시스템 아키텍처 (Architecture)

| 단계 | 수행 작업 | 사용 기술 |
| :--- | :--- | :--- |
| **Data Scraping** | 40여 명의 전문가 피드에서 최신 트윗 및 이미지 수집 | `Feedparser`, `BeautifulSoup`, `Nitter` |
| **Preprocessing** | 데이터 정제 및 ID 부여, 이미지 바이트 변환 | `Requests`, `Python` |
| **AI Analysis (1차)** | 180개 데이터 일괄 분석 및 시장 인사이트 리포트 생성 | `Gemini 3.1 Flash Lite` (Multimodal) |

| **Automation** | 정해진 시간에 코드 실행 및 환경 변수 관리 | `GitHub Actions`, `YAML` |
| **Delivery** | VIP/일반 사용자 등급별 이메일 발송 | `SMTP`, `Markdown`, `Email MIME` |

## 3. 기술적 도전과 문제 해결 (Problem Solving)

### A. 확장성을 고려한 데이터 소스 분리
* **문제:** Twikit 등 비공식 API를 통한 스크래핑을 시도(cookies.json 세션 유지 등)하였으나, 봇 로그인 제한 정책으로 인해 지속 가능한 데이터 파이프라인 구축이 불가함.
* **해결:** 로직 내부에 하드코딩된 리스트에 의존하는 대신, 구글 스프레드시트를 외부 DB로 분리하여 연동.
* **결과:** 에러 발생 가능성을 원천 차단하고, 코드 수정 없이도 운영자가 실시간으로 타겟 전문가 풀을 추가/삭제할 수 있는 동적 관리 아키텍처 구현.

### B. 멀티모달 대용량 처리 및 503 에러 대응
* **문제:** 이미지 180장을 포함한 대규모 컨텍스트를 한 번에 API로 전송할 때 서버 과부하(`503 Service Unavailable`) 및 타임아웃 발생.
* **해결:**
  * **지수 백오프(Exponential Backoff):** `Tenacity` 라이브러리를 활용해 에러 발생 시 대기 시간을 2배씩 늘리며 최대 10회까지 재시도하는 로직 설계.
  * **모델 이원화:** 텍스트 및 이미지가 많아 방대한 연산량이 요구되는 1차 요약에는 `Gemini-3.1-flash-lite` 모델을 사용하고, 개인 투자 포트폴리오를 리밸런싱하는 2차 요약은 한 차원 높은 추론 능력을 보여주는 `Gemini-3-flash-preview` 모델을 사용하여 안정성을 확보하고 성능을 최적화함. (OpenAI, xAI 대비 무료 제공 토큰 양이 압도적으로 많아 대용량 처리에 적합)

### C. API 과금 최적화 및 토큰 관리
* **문제:** 유료 티어(Tier 1) 전환 후 토큰 사용량에 따른 비용 급증 우려.
* **해결:**
  * **데이터 다이어트:** 2차 호출 시 불필요한 이미지 데이터를 제거하고 요약 텍스트만 전송하여 호출 비용을 50% 이상 절감.
  * **토큰 임계값 관리:** 128k 토큰 과금 구간을 고려한 데이터 수집량(인당 5개) 최적화.
  * **I/O 동기화:** 1차 요약과 2차 전략 수립 사이에 `time.sleep(60)`을 강제하여 서버의 분당 토큰 처리 한도(TPM)를 초기화함.

### D. 자동화 환경에서의 보안 및 식별 로직
* **문제:** 로컬 환경과 다른 GitHub Actions 환경에서 환경 변수 누락 및 VIP 수신자 식별 실패.
* **해결:**
  * `.yml` 워크플로우를 통한 환경 변수 매핑 및 GitHub Secrets를 활용한 민감 정보 보안 강화.
  * 문자열 정규화(`strip()`, `lower()`)를 적용한 방어적 프로그래밍으로 수신자 식별 로직 정밀화.

### E. 애자일 기반의 피처 다이어트 및 모듈화
* **과정:** 알림의 영속성을 위해 Notion API를 연동하여 기록 로직을 구현.
* **의사결정:** 실제 테스트 결과, 이메일 태그 필터링 대비 UX 상의 이점이 부족하다고 판단. 시스템 복잡도를 낮추기 위해 해당 기능을 과감히 제거.
* **해결:** 해당 코드를 완전히 삭제하지 않고 `.notion.py` 파일로 별도 분리하여, 시스템 결합도를 낮추고 추후 확장 가능성만 열어둠.

## 4. 주요 성과 및 인사이트 (Results & Impact)
* **효율성:** 기존에 인간 분석가가 수행할 경우 3~4시간 소요되던 180개 데이터 분석 및 리포트 작성을 **5분 이내로 단축**. (신규 소식이 가장 빠르게 공유되는 X 서비스를 메인 타겟으로 정해 타 플랫폼 대비 빠른 시장 예측과 대비 가능)
* **비용 최적화:** 이미지 데이터 필터링 및 토큰 관리 로직을 통해 1회 API 호출 비용을 기존 대비 **25% 수준으로 절감**.
* **안정성:** 지수 백오프 재시도 로직 추가 후, 503 에러(서버 과부하 환경)에서도 데이터 손실 및 **파이프라인 중단율을 0%대로 감소**시키는 데 성공.
* **정확도:** 멀티모달 분석을 통해 텍스트만으로 놓칠 수 있는 차트의 기술적 분석 논리를 보고서에 포함.
* **확장성:** 구글 시트를 구독자 및 트위터 계정 DB로 사용하여, 개발자가 아니더라도 분석 대상 전문가와 구독자 리스트를 쉽게 관리할 수 있는 구조 구축.
